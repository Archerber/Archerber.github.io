afl-analyze.c

用法

将afl项目编译后会得到一个afl-analyze文件，通过这个文件可以通过输入特定文件，对代测试文件进行分析
用法的一个例子:

这个文件最主要的函数为analyze和run_target函数
analyze中通过in_data的处理传递给run_target执行，run_target中根据analyze传递的参数，来进行fork+execv等操作，最后返回一个cksum(产生crashes会变化)

afl-as.c

作用(as是GNU汇编器，主要用于把汇编代码转换成二进制代码，并存放到一个object文件中。)

比较重要的函数有edit_params和add_instrumentation
相关参数

main

函数流程
1.获取环境变量AFL_INST_RATIO(该环境变量主要控制检测每个分支的概率，即插桩密度，取值为0到100%，设置为0时则只检测函数入口的跳转，而不会检测函数分支的跳转)
2.设置随机数
3.调用edit_params检查传递给as的参数
4.检查环境变量(AS_LOOP_ENV_VAR)
5.设置AS_LOOP_ENV_VAR为1
6.检查AFL_USE_ASAN和AFL_USE_MSAN环境变量
7.调用add_instrumentation对程序进行插桩
8.调用fork，执行execvp，execvp用as_params[0]来完全替换掉当前进程空间中的程序
9.等待子进程(waitpid)
10.读取AFL_KEEP_ASSEMBLY环境变量(设置该环境变量主要是为了防止 afl-as 删掉插桩后的汇编文件，设置为1则会保留插桩后的汇编文件)

edit_params

函数主要功能是根据程序运行参数等，检查并修改要传递给”as”的参数。
这个函数流程
1.设置tmp_dir和AFL_AS，若为APPLE架构，检查是否设置clang模式和afl_as
2.若clang_mode=1和afl_as为0，则设置use_clang_as，将afl_as设置为AFL_CC，未设置AFL_CC -> AFL_CXX，未设置AFL_CC -> AFL_CXX
3.为as_params开辟(argc + 32) * 8大小的空间，然后为as_params[0]赋值
4.然后设置tmp_dir和as_params参数，以及use_64bit(不支持32位apple)等
5.接下来设置modified_file(插桩后的文件)
最后得到的as_params，参考如下

add_instrumentation

函数功能为处理input_file，生成 modified_file。 在适当的地方插桩(只在代码段插桩)
函数流程
1.打开input_file和modified_file，并获取对应的文件描述符，方便操作文件
2.若不存在input_file，则将inf的文件描述符设置为stdin
3.进入while循环逐行读取input_file，MAX_LINE(8192)，
4.插桩寻找代码段，设置instr_ok = 1
5.检测off-flavor assembly（很少见，发生在 gdb 中）遇到后设置skip_csect标志位，不插桩
6.跳过intel汇编，设置skip_intel
7.跳过 ad-hoc asm
8.寻找跳转指令(jne,je,jle……)
判断条件和插桩代码

instr_ok设置

条件跳转插桩相关代码(inst_ratio表示插桩密度，默认值为100，asan下为30左右)

instr_ok：用于判断text段，可能需要插桩

afl-fuzz.c

运行afl-fuzz重要的参数和模式

感觉分析的话从main好入手一点

main

main函数流程
1.从命令行读取参数进行相关设置，如input_file等(save_cmdline)
2.设置程序信号处理+检查asan选项+获取环境变量，设置对应的变量(获取cpu核心数，检查core_pattern，检查是否在tty，检查cpu_governor)
3.设置后处理器，共享内存，设置setup_dirs_fds(输出目录,quene，crashes等文件都是在这里面创建)等
4.读入测试案例，排队进行测试(read_testcases)，然后加载自动生成的提取出来的词典token
5.为输入文件创建hard link
6.判断extras_dir，调用load_extras(从extras_dir读取extras到extras数组里，并按size排序)
7.根据timeout_given，调用find_timeout函数(设置exec_tmout)，调用detect_file_args(判断参数里是否有‘@@’，有替换为out_dir/.cur_input)
8.然后是setup_stdio_file(out_file为空，没有使用-f选项时，删除原有的out/.cur_input文件，创建一个新的out/.cur_input)
9.调用check_binary(检查执行程序的路径是否存在)，根据qemu_mode=1调用get_qemu_argv(获取qemu参数)
10.执行perform_dry_run(执行所有的测试用例，以检查是否按预期工作)
11.调用cull_queue(精简queue)，调用show_init_stats(在处理输入目录的末尾显示统计信息，以及一堆警告,以及几个硬编码的常量)
12.调用find_start_position函数(恢复fuzz时，找到开始的队列位置)，然后调用write_stats_file函数(更新统计信息文件以进行无人值守的监视，这个与fuzz界面上的信息相关)
13.调用save_auto函数(保存自动生成的extras)，然后进入while的主循环，进而开始fuzz
while循环流程：
1.首先调用call_queue精简队列，然后获取queue_cur
2.queue_cur为0时，表示所有queue都被执行一遍，queue_cycle(表示所有queue被完整执行的次数)++，然后设置current_entry为0，queue_cur为首要元素
3.判断seek_to是否为0，为0时，从seek_to指定的queue处开始执行
4.刷新显示状态(show_stats)
5.然后通过queued_paths和prev_queued判断是否有新发现，如果use_splicing为1，则cycles_wo_finds++，否则设置use_splicing为1，接下来使用splice重组queue中的test
6.判断queue完整执行的次数，若为1，则调用sync_fuzzers(读取其他sync文件夹下的queue文件，然后保存到自己的queue里)
7.执行fuzz_one(对queue_cur进行测试)，并将结果返回(不执行为1否则为0)给skipped_fuzz
8.接下来若skpped_fuzz为0，且sync_id存在，则sync_interval_cnt++，并且sync_interval_cnt为SYNC_INTERVAL的倍数，则调用sync_fuzzers(同步queue)
9.queue_cur = queue_cur->next;current_entry++;测试下一个queue

fuzz_one

1.对一些变量进行定义和初始化
2.判断pending_favored>0，且queue_cur被fuzz过，并且没被标记favored，则有99%概率直接返回1
3.如果pending_favored=0，且queued_paths>10，未被标记favored
1).如果queue_cycle > 1，并且queue_cur没有被fuzz过，则有%75的概率返回1
2).否则有95%的概率返回1
4.打开queue_cur->fname，获取queue_cur的长度(len = queue_cur->len)，将文件映射进内存，然后将mmap得到的地址返回到orig_in和in_buf
5.分配len的空间，然后将地址给out_buf，获取queue_cur的深度(queue_cur->depth)
然后可以分阶段分析

CALIBRATION(仅当之前失败时)

当效准错误(queue_cur->cal_failed)<3时，使用calibrate_case函数再次校准

TRIMMING(修剪)

通过queue_cur->trim_done，判断该case是否经过trim，未经过trim则调用trim_case进行修剪，设置queue_cur->trim_done = 1，然后将queue_cur->len给len，最后通过memcpy函数，
将in_buf中的内容复制到out_buf

PERFORMANCE SCORE

使用calculate_score函数计算得分(orig_perf = perf_score = calculate_score(queue_cur);)
此时根据skip_deterministic、queue_cur->fuzzed、queue_cur->passed_det来确定是否跳转到havoc_stage(havoc模式)，若其中有一项为1则跳转到havoc_stage
如果 exec 路径校验和超出范围，也会直接跳转到havoc_stage，若为发生跳转则设置doing_det为1

例如size,flags等)，在判断时，若翻转一字节程序的执行路径变化，则将该字节标记为1，表示有效，否则标记为0，表示无效。
在之后的变异会跳过无效bytes，这样可以大大节省资源，在dumb模式和文件非常小时，每个字节都被标记

ARITHMETIC INC/DEC

主要代码如下

主要对文件的bytes进行加减，从而变异文件，加减的范围根据ARITH_MAX(35)控制，范围从+1、-1、+2、-2、+3、-3一直到+35, -35，然后会跳过加减之后在bitflip变异中出现的bytes，
之后会对多个字节进行加减变异操作，包括2bytes和4bytes，由于大于两个字节后，存在大端和小端情况，afl会对这两种情况都进行变异

DICTIONARY STUFF

这里分为3个阶段：
1.user extras(over)，这一部分由用户提供extras[]在load_extras函数中设置，然后替换out_bf(即文件中的内容)并执行
2.user extras(insert)，这一部分插入extras[]，这里定义了一个ex_tmp，然后再将ex_tmp传入common_fuzz_stuff执行
3.auto extras(over)，这里替换的extras为a_extras，在auto_save中生成
用户提供的token在这里表现为extras[]，为全局变量，通过-x选项选定(仅在load_extras函数中生成)，没有则跳过这一阶段

RANDOM HAVOC

这一部分变异属于非确定性变异(没有规律的变异规则)，domb mode和从fuzz则从这一阶段开始，而主fuzz则是先完成确定性变异，再到这一部分

SPLICING

当队列执行完一次后，没有发现则进入SPLICING模式，是对文件的最后的变异操作，使用当前输入文件，并随机选取另一个输入文件，将这两个文件在某个偏移处拼接在一起，然后使用havoc来进行变异
操作顺序，如下：
1.判断in_buf是否改变，改变则使用orig_in恢复
2.选择一个随机队列条目并查找，找到后，将testcase输入到new_buf，未找到重新SPLICING
3.找到一个合适的拼接位置，位置介于第一个和最后一个不同的字节之间
4.在第一个和最后一个不同字之间随机选取一个位置
5.使用memcpy进行拼接，然后进行havoc_stage
