堆

主分配区(mian_arena)

当申请空间时，如果能分配到空间，则会创建主分配区，主分配区的空间可以来自mmp和heap，每次分配内存必须对该空间加锁，分配完成释放锁

bins

ptmalloc通过bins将chunk联系起来，并通过不同的索引使用不同的chunk，bins中包括 unsorted bins、small bins，largebins，这些bins中储存的chunk都是通过双向链表联系起来的

fastbins

当malloc申请的内存小于64B(32位)，128B(64位)时，会使用fastbin，fastbin通过单向链表将bins连起来，fastbins最多支持10个bins，从8B到80B，fastbins中的chunk的p位始终为1，保证不回被合并，只有当某个特定的时候才会被合并
fastbin中堆的大小范围为0x10-0x80，并通过单向链表连接

smallbins

索引从2到63，共62个bin，为循环双向链表，chunk的大小从16B-63B，每个bins中存放的chunk大小固定，64位最大为0x400

largebins

32位大于512b,64位大于1024b，属于largebin，largebin总共包括63个bin，每组bin的数量依次为32、16、8、4、2、1
每组bin的公差为64b、512b、4096b、32768b、262144b，
chunk_size = 512 + 64 * index(第一个bin的大小)
chunk_size = 512 + 64 * 32 + 512 * index(第二个bin的大小，要算上前一个bin，后面依次类推)
见下图

largebin的结构中包含fd_nextchunk以及bk_nextsize(指向别的堆头的指针，不是大小)
fd_nextchunk和bk_nextsize的作用是连接同一个bin中不同大小的bin

unsortbins

unsortedbins只有一个bin，chunk为乱序，chunk被分割之后如果剩下的部分大于64B就被放到这里(当释放一个不属于fastbins的chunk，chunk会先到这里)，可以理解为回收chunk的缓存区，当unsortbin中只有一个bin时，他的fd以及bk会指向main_arena，如果未从fastbin中找到合适的chunk，会从unsorted bin中找起

top chunk

当free的chunk大于64B时，如果与topchunk相邻，该chunk会与topchunk合并(释放fastbin不用担心，必不可能与topchunk相邻)

堆块合并

当free掉两个空闲且相邻的chunk(非fastbin)之后，这两个chunk会合并成一个大chunk，然后放入unsorted bin中，fastbin中的pre_in_use始终为1，不会发生堆块合并

关于free时chunk的去向：

当我们free掉一个chunk时，系统会根据这个chunk的大小来判断属于哪种类型的chunk，如果属于fastbin，则将它放入fastbin中，不属于则触发unlink，先合并物理低地址块再合并物理高地址块，然后放入unsorted bin中

unlink操作：

如图
操作为：
FD=P->FD,BK=P->BK,FD->BK=BK,BK->FD=FD
如果我们可以通过堆溢出修改p中的fd以及bk，则我们可以通过unlink来实现任意地址写
fd修改为 P -> fd = target - 0x12 , P ->bk = expect value , 在没有保护的情况下，有
FD = P -> fd = target - 0x12 , BK = P -> bk = expect value

  • ( target - 0x12 + 0x12 ) = BK = expect value
    如果 target 设置为 free 的 got表，expect value 设置为system，这样就可以将
    free的got表改为system
  • ( expect value + 0x8 ) = target - 0x12
    expect value + 0x8 需要设置为可写地址
    上面的情况只适用于没有保护的情况，当有保护之后，情况就要复杂一些
    保护为：

1.检查大小
检查可以合并的chunk大小，第一处是，当前chunk的presize(包括head)与被合并
chunk的大小是否对应，第二处，是被合并的chunk的next chunk’s presize
2.检查链表完整性
即 FD -> bk == P 以及 BK -> fd == p

检查的绕过：

当用户申请chunk的流程：
先在fastbin中寻找chunk，如果没找到则从small bin(判断大小是不是小于512B)中寻找，这时再没找到就会从unsorted bin中寻找，此时，unsorted bin中有属于smallbin中的chunk时，就会将chunk切割，然后用户就能得到chunk，并且得到的chunk的fd是切割前大chunk的fd

tcache机制介绍

libc-2.26及以后加入了tcache，用以加快chunk的分配速度，free一个chunk时，会先尝试放入tcachebins中，也因此要使chunk到
fastbin中，应先free7个同样大小的chunk，bin每一个tcache最多包含7个块，
tcache中chunk在32位从12-512bytes,64位是24-1024bytes需要超过0x400才能进unsortbin
libc-2.26 - libc-2.27 关于tcache的检查很少，可以任意的doublefree，malloc，此外tcache的表头有一个小小的漏洞，
当一个chunk被free两次之后，此时bins里的下标为2，此时再malloc 3个对应大小的堆，下标就会为-1，而此时再free掉这个堆就会放到
unsortedbin中
libc-2.29中增加了对tcache doublefree 以及unsorted bin attack 的检测
tcache 的 entry处添加了一个key
